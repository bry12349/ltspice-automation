# Interview Notes

## One-Minute Project Explanation

This project is a local Codex plugin for LTspice automation. It takes a constrained natural-language request, generates a visible LTspice `.asc` schematic for an RC low-pass step-response circuit, runs LTspice in batch mode, parses `.log` and `.meas` results, and checks the result against first-order RC theory.

The core idea is not "AI magically designs any circuit." The stronger engineering point is that the workflow is bounded, deterministic, and verified. It uses AI-assisted input as the front end, but relies on LTspice simulation and measured outputs before treating the generated circuit as valid.

## RC Circuit Theory Points

For an RC low-pass circuit with a step input:

```text
tau = R * C
Vout(t) = Vin * (1 - exp(-t / tau))
```

Important points:

- At `1 tau`, output is about `63.2%` of final value.
- At `5 tau`, output is about `99.3%` of final value.
- Larger `R` or `C` increases tau and slows the response.
- Smaller `R` or `C` decreases tau and speeds the response.

Default example:

```text
R = 1k ohm
C = 1uF
tau = 1 ms
Vin = 1 V
Vout(1 ms) = 1 * (1 - exp(-1)) = 0.632121 V
```

Phase 3 improvement:

- The tool no longer assumes `1 ms` always means `1 tau`.
- For `R=2k` and `C=1uF`, tau is `2 ms`, so measurements move to `2 ms` and `10 ms`.
- The tau-cross target scales with input voltage, so a 5 V step uses about `3.1606 V`.

## LTspice File Types

### `.asc`

LTspice schematic file. It stores the visible circuit layout: symbols, wires, labels, and text directives. This project generates `.asc` because users can open and inspect the schematic graphically.

### `.cir`

SPICE circuit netlist. It is text-first and does not preserve a visible schematic layout. It is useful for explicit or custom simulations.

### `.net`

Netlist generated by LTspice from a schematic. When an `.asc` is simulated, LTspice converts it into a netlist internally. The `.net` file is useful for checking what circuit LTspice actually simulated.

### `.log`

Simulation log. It contains simulator metadata, warnings, errors, and `.meas` results. This project parses `.log` to determine whether the simulation succeeded and to extract measurements.

### `.raw`

Waveform data file. It stores transient, AC, or operating-point result data for plotting and deeper analysis. This project currently does not parse `.raw`.

### `.meas`

Not usually a separate file in this project. `.meas` is a SPICE directive placed inside `.asc` or `.cir`. LTspice evaluates it during simulation and writes the result into the `.log`.

## Common Interview Questions

### Why start with RC low-pass instead of arbitrary circuits?

Because RC low-pass is small enough to verify thoroughly. It has clear theory, predictable transient behavior, and simple `.meas` targets. Starting narrow made it possible to build a reliable loop: generate, simulate, parse, compare, and test.

### What makes this more than just generating a text file?

The workflow runs LTspice and reads the simulator output. A generated schematic is not considered enough. The project checks `.log` errors and `.meas` results, then compares values against expected RC behavior.

### How do you know the circuit is correct?

For the default case, theory gives `tau = R*C = 1 ms` and `Vout(1 tau) = 0.632121 V`. LTspice reports about `0.631937 V` at 1 ms, which is within a small numerical error. The smoke test checks this automatically.

### What was the most important bug you fixed?

The original measurement logic assumed every RC circuit had `tau = 1 ms` and `Vin = 1 V`. That was correct only for the default example. I changed the code so measurement times come from `R*C`, and the tau-cross voltage target scales with input voltage.

### Why generate `.asc` instead of only `.cir`?

`.cir` is easier to generate, but `.asc` is more useful for LTspice users because it gives a visible schematic. A user or interviewer can open it and inspect whether the circuit topology makes sense.

### What are the limitations?

The natural-language schematic generator currently supports constrained RC low-pass and RL step-response workflows. GUI opening is macOS-specific. Log parsing is useful but still basic. RLC, Buck converter, and parameter sweeps are future work.

### How did you extend this to RL?

I first wrote the RL template design, then added a constrained series RL `.asc` template with `tau = L/R`, final current `I_final = Vin/R`, `.tran`, and `.meas` directives for current at 1 tau and 5 tau. I verified the generated LTspice netlist and fixed the inductor symbol orientation so `I(L1)` measured the expected current.

### How would you extend this to Buck converter?

I would not jump directly from RC to Buck converter. Buck converters need switching behavior, duty cycle, inductor ripple, diode/MOSFET models, load conditions, and stability concerns. I would first add RL and RLC templates, then implement a constrained Buck template with clear assumptions and tests.

### What did you learn from this project?

The biggest lesson is that AI-assisted engineering tools need hard verification boundaries. Natural language can help create the initial artifact, but simulation, measurements, theory checks, and tests are what make the output trustworthy.
